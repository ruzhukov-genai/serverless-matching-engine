## E2E Timestamp Fix (15:06 UTC)

**Bug found:** `persisted_at` was set via `NOW()` in Postgres (PG server clock), while `received_at` and `matched_at` used `Utc::now()` in Rust (Lambda clock). Two different clocks ‚Üí E2E deltas were meaningless.

**Fix:** Set `persisted_at = Utc::now()` in Rust before the INSERT, bind it as `$18`. Applied to both:
- `crates/gateway/src/worker.rs` (inline mode / gateway Lambda)
- `crates/worker-lambda/src/main.rs` (queue drain / worker Lambda)

**Post-fix benchmark (same-clock, trustworthy):**

### T02 Failure: Cache Update Bug

Smoke test T02 (limit sell rests on book) consistently fails at AWS:
- Order placed at price 77603 (high, out of immediate match range)
- Order persists to DB with `status=New` ‚úÖ
- But Valkey orderbook cache is **not updated** ‚Äî GET /api/orderbook returns stale data ‚ùå
- Root cause: `update_cache_after_processing()` is non-fatal and silently fails under connection load

This is the same Valkey contention issue from benchmarks ‚Äî concurrent order processing + Lua script execution bottlenecks the cache update path.

**Next priority:** Fix cache update to be reliable (not fire-and-forget). Options:
1. Retry with exponential backoff
2. Move cache update to the DB transaction boundary (persist + cache atomic)
3. Use Valkey pipeline for batch updates (reduce round-trips)
4. Add monitoring/alerting for failed cache updates

## Valkey Cache Fixes (15:57-16:35 UTC)

### Root Cause Chain (3 bugs, all fixed)

**Bug 1: Worker Valkey pool exhaustion**
- `get_state()` created pool with `max_size=2`
- `update_cache_after_processing()` called `set_and_publish()` N times, each doing `pool.get()`
- Under load: pool exhausted ‚Üí silent failure ‚Üí stale cache
- Fix: `set_and_publish_batch()` ‚Äî single pool.get() + pipeline for all keys; pool bumped to 5

**Bug 2: reset_all / truncate_orders wrong Valkey key names**
- Both commands DEL'd `orderbook:{pair}:bids/asks` but actual keys are `book:{pair}:bids/asks`
- Also not flushing `cache:orderbook`, `cache:trades`, `cache:ticker`
- Fix: use correct key names, also DEL all cache: keys

**Bug 3: get_orderbook served in-memory cache (stale across Lambda instances)**
- `AppState.cache` is in-process memory, updated via Valkey Pub/Sub
- Multi-Lambda: if PUBLISH from processing-instance didn't reach serving-instance ‚Üí stale forever
- Fix: `get_orderbook` reads directly from Valkey `GET cache:orderbook:{pair}` first
- In-memory cache kept as fallback + WS fan-out only

**Bug 4 (SAM build cache): Previous deploys didn't update Lambda code**
- SAM detected no Docker layer hash change ‚Üí skipped Lambda update
- Fix: `rm -rf .aws-sam` before `sam build` to force fresh layer build
- ‚öÝÔ∏è Remember: if code changes don't appear to deploy, delete .aws-sam and rebuild
